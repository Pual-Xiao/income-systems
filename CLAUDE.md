# CLAUDE.md — 贡献评估模型

用同一把 12 维度的尺子，衡量一个人在任何一件事里的贡献。

## 解耦架构

项目由三个独立模块组成，改一个不动另外两个：

- **模块 1 · 事件定义**：输入（人 + 做了什么 + 花了多久）→ 输出（结构化事件描述）
- **模块 2 · 衡量尺子（12 维度）**：输入（事件描述）→ 输出（维度得分 + 相关性 + 确定性）
- **模块 3 · 计算工具**：输入（得分 + 相关性 + 确定性 + 时间）→ 输出（有效贡献小时 + 排名）。内部含因果三层：L1 事件本身 → L2 直接影响 → L3 涟漪扩散

> 注意区分：模块 1/2/3 是**软件架构**解耦。L1/L2/L3 是模块 3 内部的**因果模型**。两套概念不是同一套。

## Layer 1 · 事件定义

### 核心概念

事情可以无限嵌套拆分。拆分（事件的结构）和分配（谁来做）是两层：大事件拆成小事件，再分配给人/工具执行。贡献在执行中产生。

### 事件基本要素

- **什么事**（具体做什么）、**父事件**（属于哪件更大的事）
- 执行层才出现：**执行者**、**耗时**、**具体动作**

### 事件层级与结算时机

```
父事件（项目）→ 子事件（阶段任务）→ 叶子行为（具体动作）
```

- 父事件完结 → 回溯评估所有子行为的因果贡献 → 评分锁定
- 父事件进行中 → 子行为评分只能是暂定的
- 叶子事件（父事件 = 自身）→ 行为完成即完结 → 即时结算

### 分类协议（LLM 操作规则）

#### 零、冷启动（新对话开始时执行一次）

```
1. get_ongoing_events(person_id)    → 有哪些进行中的事件
2. get_pending_pool(person_id)      → 待分配池有什么
3. 有进行中事件 → 用户新行为优先归入相关事件
4. 有待分配项 → 提醒用户
5. 都为空 → 全新状态，等用户输入
```

#### 一、父事件是否明确？—— 三条规则（按顺序判断，命中即停）

**规则1**：行为本身是叶子事件？
- 行为完成 = 结果立即可见，不依赖更大项目
- 判定：父事件 = 自身（新建同名事件），即时结算

**规则2**：用户明确说了父事件，或可从对话上下文推断？
- 用户说了"这是为XX项目做的" → 父事件明确
- 对话上下文在讨论某个已知项目 → 父事件可推断
- 以上不满足 → 进入规则3

**规则3**：以上都不满足 → 父事件不明确 → `add_pending()` 写入待分配池

#### 二、待分配池归类——四条途径

- **A**：用户后续说明用途 → `classify_pending()` 归类
- **B**：资产被某事件消耗 → 自然流向使用处
- **C**：资产长期未消耗 → 留在 pending，定期检查；确定永不被消耗 → 纯浪费
- **D**：多事件消耗同一资产（资产复用）→ `allocate()` 分配给多个父事件（对应复利维度）

#### 三、事件完结判断

- 叶子事件：行为完成 = 完结
- 子事件：所有子行为完成 + 产出可验证
- 父事件：所有子任务完成 + 最终目标达成或终止

完结时调用 `complete_event(person_id, event_id, outcome)`。

## Layer 2 · 12 维度衡量尺子

### 尺子定义

| 维度 | 权重 | 正向表现 | 负向表现 |
|------|------|----------|----------|
| 创意 | 1.2 | 提出新方案、新视角、新思路 | 抄袭、扼杀创意、思维僵化 |
| 规划 | 1.0 | 拆解任务、制定可行路径 | 计划混乱、方向错误 |
| 决策 | 1.3 | 做出正确判断和选择 | 优柔寡断、错误判断 |
| 行动 | 1.5 | 落地执行、产出结果 | 拖延、半途而废 |
| 技能 | 1.0 | 运用专业能力完成工作 | 能力不足、粗制滥造 |
| 协作 | 1.1 | 沟通配合、帮助他人 | 冲突、信息封闭 |
| 领导 | 1.2 | 组织激励、引导方向 | 独断、打压 |
| 学习 | 0.8 | 获取新知、提升能力 | 拒绝学习、固步自封 |
| 维护 | 0.9 | 修复问题、保持稳定 | 制造隐患、留下烂摊子 |
| 影响力 | 1.4 | 正面效应扩散 | 负面传播、损害声誉 |
| 创新 | 1.3 | 新方法解决老问题 | 为创新而创新、破坏有效系统 |
| 复利 | 1.5 | 持续复利回报（工具/知识/体系积累） | 一次性消耗、透支未来 |

### 评分规则：四层判断 + 三种结算

**前置**：时间即公平基准——L1 = 相关维度权重和 × 时间，保证每分钟都被计入。

对每个维度，逐层判断：

```
第一层 · 相关性 → 跳过 / 相关
第二层 · 效果方向 → 正向(+1~+5) / 负向(-5~-1) / 零效果(0) / 不可评
第三层 · 零效果类型 → 潜伏价值 / 纯浪费
第四层 · 确定性 → 确定 / 模糊
```

#### 第一层：相关性

| 判定 | L1 | L2 |
|------|----|----|
| **跳过**：维度与此事无关 | 不计入权重 | 不计入 |
| **相关**：维度与此事有关 | 计入权重和 | 进入第二层 |

#### 第二层：效果方向（因果贡献，非行为好坏）

判断的是**此行为对父事件最终结果的因果贡献**。同一行为在不同父事件中可完全不同。

- **正向 (+1~+5)**：推动了父事件向好的结果发展
- **负向 (-5~-1)**：阻碍/偏离了父事件的最终结果（父事件完结前不存在）
- **零效果 (0)**：对父事件结果无实质因果影响
- **不可评**：父事件未完结（暂定），或行为未分配给任何父事件（延期）

**执行质量 ≠ 因果贡献**：
- 质量高 + 结果需要 → 正分 │ 质量高 + 结果不需要 → 0分
- 质量一般 + 关键决策对 → 正分 │ 质量高 + 带偏项目 → 负分

#### 第三层：零效果类型

| 类型 | 含义 | 例子 |
|------|------|------|
| **潜伏价值** | 花了时间当前没用，未来可能有价值 | 检索1h没找到信息，但积累了经验 |
| **纯浪费** | 花了时间确实没产出，也不会产生后续价值 | 方向错了白做、资产买了没用 |

两种 L2 均为 0。区别：潜伏价值可能在「学习」「复利」等维度有额外正分。评分可修正。

#### 第四层：确定性

| 判定 | L2 计算 |
|------|---------|
| **确定**（判断明确）| weight × score × 1.0 |
| **模糊**（不确定方向或程度）| weight × score × 0.5 |

### 三种结算模式

| 模式 | 场景 | 行为完成时 | 父事件完结时 |
|------|------|-----------|-------------|
| **即时结算** | 叶子事件（喝水解渴、修好bug）| 评分锁定 | — |
| **暂定结算** | 子事件（为进行中项目做调研）| 暂定评分 | 回溯修正→锁定 |
| **延期结算** | 父事件不明确（买水用途未定）| 只记录不评分 | 归类→暂定→锁定 |

### 资产与待分配投入

行为发生 → 时间花费（L1 锁定）→ 产出资产 → 进入待分配池 → 分配给父事件 → 父事件完结 → 评估因果贡献 → 锁定

- **资产可复用**：一次学习可用于多个项目，每个独立结算（对应复利维度）
- **资产可过期**：未被消耗 = 纯浪费（L1 不受影响，时间已花）
- **L1 在行为发生时锁定**：不论资产最终如何结算

## Layer 3 · 计算工具

### 时间货币公式

**1 贡献点 ≡ 1 有效工作小时**

```
有效贡献小时 = A × (L1 / total_weight) × l2_factor × l3_factor

L1 = Σ(相关维度权重) × 小时
l_factor = 1 + L2(或L3) / (对应权重和 × quality_scale)
```

全部维度 0 分 → 系数 1.0（时间等值转换）；正分放大、负分缩水、极端负向可为负。

### 参数

| 参数 | 值 | 含义 |
|------|-----|------|
| A | 1.0 | 总体校准系数 |
| quality_scale | 2.5 | 均分±2.5→系数0~2；均分-5→系数-1 |
| fuzzy_factor | 0.5 | 模糊维度的质量折扣 |

### 排名映射

| 名称 | 累计有效小时 | 含义 |
|------|-------------|------|
| 破坏者 | < 0 | 净贡献为负 |
| 新手 | 0 ~ 8 | 不到一天 |
| 实践者 | 8 ~ 40 | 一天到一周 |
| 贡献者 | 40 ~ 200 | 一周到一个月 |
| 创造者 | 200 ~ 1000 | 一个月到半年 |
| 引领者 | 1000+ | 半年以上 |

## 三层接口

```
Layer 1 → Layer 2：结构化事件描述（谁、做了什么、花了多久、父事件）
Layer 2 → Layer 3：维度得分 + 相关性 + 确定性 + 耗时
Layer 3 → 输出：有效贡献小时 + 排名
```

| 文件 | 所属层 | 职责 |
|------|--------|------|
| （待建：事件描述规范）| Layer 1 | 事件格式和拆分规则 |
| model.json | Layer 2 + 3 参数 | 维度定义 + 权重 + 公式参数 + 排名 |

| ledger.py | 数据管理 | 账本操作：记录→分配→结算→查询，调用 evaluator.py |
| data/{person}/pending.json | 数据 | 待分配池：父事件不明确的行为 |
| data/{person}/ledger.json | 数据 | 个人账本：已分配行为 + 事件 + 结算状态 |
| flash-points.md | 记录 | 对话演化关键节点 |

### 多用户数据架构

```
data/{person_id}/
  ├── pending.json   ← 待分配池
  └── ledger.json    ← 个人账本
```

- 一个文件夹 = 一个人，数据物理隔离。新用户首次调用自动创建
- pending.json 和 ledger.json 分开：没归类的行为不进账本
- `ledger.py` 所有函数第一个参数为 `person_id`

## 评分流程（端到端）

```
0. 冷启动：get_ongoing_events() + get_pending_pool() → 了解当前状态
1. 用户描述一件事（做了什么、花了多久、属于哪件更大的事）
2. 按分类协议判断父事件：
   - 规则1（叶子事件）→ 新建同名事件 → 跳步骤4
   - 规则2（父事件明确/可推断）→ 使用已有事件ID → 跳步骤4
   - 规则3（父事件不明确）→ add_pending() → 结束
3. 待分配池定期 review，classify_pending() 归类 → 进入步骤4
4. add_behavior(person_id, ..., parent_event_id=eid) 写入账本
5. 按 Layer 2 四层判断，对每个相关维度打分
6. settle() → 父事件完结→锁定；未完结→暂定
7. 父事件完结时 complete_event() → 回溯修正未锁定评分
8. compute_contribution() → 单次有效贡献小时 + 排名
9. compute_event_total() → 事件累计贡献（跨会话）
10. compute_person_total() → 个人累计总分 + 排名
```
